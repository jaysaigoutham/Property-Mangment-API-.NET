Build a frontend web project for a property listing marketplace that connects to an existing .NET 10 microservices backend.

Backend summary:
- Primary API gateway URL: http://localhost:8080
- Use the gateway URL for normal frontend API calls.
- Direct service ports are only for debugging:
  - Identity: http://localhost:8081
  - Listings: http://localhost:8082
  - Media: http://localhost:8083
  - Engagement: http://localhost:8084
  - Notifications: http://localhost:8085
  - Admin: http://localhost:8086
  - Payments: http://localhost:8087
- Existing Postman collection reference: ../PropertyMarketplace.postman_collection.json

Frontend goal:
Create an actual usable web application, not a landing page. The app should let buyers search and engage with listings, agents manage property listings and paid ad checkout, and admins moderate and manage marketplace operations.

Recommended stack:
- React + Vite + TypeScript, unless the current project uses a different stack.
- Use a clean responsive layout.
- Use environment config, for example:
  - VITE_API_BASE_URL=http://localhost:8080
- Centralize API calls in a typed API client.
- Use form validation and clear loading/error/empty states.

Authentication and authorization:
- Backend auth uses JWT bearer tokens.
- Login/register responses include an access token.
- Store the JWT securely on the client side.
- Send protected API calls with:
  Authorization: Bearer <token>
- Supported roles:
  - buyer
  - agent
  - admin
- The UI must separate navigation and permissions by role.
- Buyers should not see agent listing management or admin tools.
- Agents should see listing management, media management, and paid ad checkout.
- Admins should see marketplace moderation and payment administration screens.

Required pages and flows:

1. Auth pages
- Login page.
- Register page with role selection for buyer or agent.
- Profile page for the logged-in user.
- Logout action.

2. Public marketplace pages
- Listing search page.
- Listing filters:
  - q
  - city
  - country
  - propertyType
  - minPrice
  - maxPrice
  - bedrooms
  - page
  - pageSize
- Listing details page.
- Public listing detail should show listing information, amenities, images, agent information where available, reviews, and actions based on role.

3. Buyer dashboard
- Saved favorites.
- Saved searches.
- Submitted inquiries.
- Ability to favorite/unfavorite listings.
- Ability to create saved searches.
- Ability to send an inquiry for a listing.
- Ability to create a review for an agent/listing.

4. Agent dashboard
- My listings.
- Create listing.
- Edit listing.
- Delete listing.
- Listing status display:
  - Draft
  - PendingApproval
  - Approved
  - Rejected
- Submit listing for approval.
- Important: submitting a listing requires an active completed paid listing ad entitlement. If the backend returns payment required, route the agent to checkout.

5. Media manager
- Generate upload URL for listing images.
- Upload image file using the returned upload URL.
- Display listing images.
- Reorder images.
- Remove images.

6. Paid ad checkout
- Show available ad packages from the backend.
- Allow agent to choose an ad package for a draft listing.
- Allow optional promo code entry before checkout.
- Show checkout preview:
  - original amount
  - discount amount
  - final amount
  - currency
- Create checkout using Stripe or PayHere.
- For Stripe, redirect the user to the hosted checkout URL returned by the backend.
- For PayHere, use the hosted checkout URL and form fields returned by the backend.
- After payment, show a payment status/check page using checkout details.
- After a completed payment entitlement exists, allow listing submission for approval.
- Never calculate final payable amount on the frontend; always ask the backend for preview/create checkout.

7. Admin dashboard
- Service summary/status.
- User list.
- Listing moderation:
  - approve listing
  - reject listing
- Review moderation:
  - pending reviews
  - publish review
  - reject review
- Notifications list.
- Notifications by topic.
- Payment administration:
  - list/create/update/delete ad packages
  - list/create/update/delete promo codes
  - list payment checkouts
- Admin screens require an admin JWT.

Expected frontend behavior:
- Use optimistic UI only where safe; otherwise refresh from the backend after writes.
- Display backend validation errors clearly.
- Handle 401 by redirecting to login.
- Handle 403 by showing a not-authorized view.
- Handle listing submit payment requirement by taking the agent to checkout.
- Keep admin functionality hidden unless the logged-in user role is admin.
- Keep payment provider secrets out of the frontend. The frontend only receives hosted checkout URLs or PayHere form fields from the backend.
- Do not build fake/mock-only pages. Connect screens to the backend API client.

Deliverables:
- Complete frontend project with install/run instructions.
- Environment file example.
- Typed API client or equivalent API service layer.
- Responsive pages for buyer, agent, and admin workflows.
- Clear README explaining how to run the frontend with the backend gateway.
